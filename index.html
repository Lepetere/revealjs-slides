Entwurf einer leicht verständlichen API für Clarango

- Spannungsfeld zwischen klassischen Datenbanktreibern und „clojuresquen“ Methoden - bzw. zwischen einer zustandshaften Datenbank und zusandslosem, funktionalem Design - untersuchte Treiber

Probleme und grundsätzliche Fragen

konsistente API mit ähnlichen Methoden

- Einteilung in Namespaces (vs. Gliederung der ArangoDB HTTP API) - Methodenbenennung - Kaschierung von Inkonsistenten der ArangoDB HTTP-API

Probleme und grundsätzliche Fragen

flexible Methoden

- sollten mit wenigen Parametern aufgerufen werden können (dann Verwendung von Default-Werten) - alternativ aber auch Übergabe von mehreren Parametern möglich für manuelle Konfiguration - gleichzeitig sollte es aber nicht verwirren und eine einfache Dokumentation möglich sein

							(defmulti handle-error
  "Handle an error trying to access a resource."
  (fn [error] (type error)))
(defmethod handle-error java.net.ConnectException
  [error]
  (throw (Exception. (build-connection-exception-string error))))
(defmethod handle-error clojure.lang.ExceptionInfo
  [error]
  (throw (Exception. (build-server-exception-string error))))
(defmethod handle-error :default
  [error]
  (throw error))
