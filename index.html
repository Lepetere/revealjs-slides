Gestaltungsprobleme, grundsätzliche Fragen

Probleme und grundsätzliche Fragen

Verbindungsdatenspeicherung

- REST/HTTP-Schnittstelle ist zustandslos - klassische, zustandsbehaftete DB-Verbindung kommt hier nicht vor - trotzdem sollte es eine Möglichkeit geben, Verbindungsdaten dauerhaft zu speichern, damit diese nicht permanent angegeben bzw. in einer selbstverwalteten Variable zwischengespeichert werden müssen - daher Speicherung in einer dynamischen Variable im Namespace core, ähnlich wie bei den untersuchten DB-Clients Monger, Elastisch und Neocons

Probleme und grundsätzliche Fragen

Entwurf einer leicht verständlichen API für Clarango

- Spannungsfeld zwischen klassischen Datenbanktreibern und „clojuresquen“ Methoden - bzw. zwischen einer zustandshaften Datenbank und zusandslosem, funktionalem Design - Orientierung an existierenden Clojure (NoSQL) Treibern

Probleme und grundsätzliche Fragen

konsistente API mit ähnlichen Methoden

- Schaffung einer einheitlichen Funktionsschnittstelle - Kaschierung von Inkonsistentien der ArangoDB HTTP-API - Einteilung in Namespaces (vs. Gliederung der ArangoDB HTTP API)

Probleme und grundsätzliche Fragen

flexible Methoden

- sollten mit wenigen Parametern aufgerufen werden können (dann Verwendung von Default-Werten) - alternativ aber auch Übergabe von mehreren Parametern möglich für manuelle Konfiguration - gleichzeitig sollte es aber nicht verwirren und eine einfache Dokumentation möglich sein

Error-Handling

							(defmulti handle-error
  "Handle an error trying to access a resource."
  (fn [error] (type error)))
(defmethod handle-error java.net.ConnectException
  [error]
  (throw (Exception. (build-connection-exception-string error))))
(defmethod handle-error clojure.lang.ExceptionInfo
  [error]
  (throw (Exception. (build-server-exception-string error))))
(defmethod handle-error :default
  [error]
  (throw error))

Ergebnisse

Architektur

  • HTTP-API Wrapper lässt sich mit sehr wenig Overhead umsetzen
  • insbesondere funktional!
  • viele wiederverwendbare Komponenten entworfen
  • alle Ressource-URIs können mit der selben Methode zusammengebaut werden (~15 LOC)
- letzter Punkt wäre ohne agile Herangehensweise schwer planbar gewesen und hätte eine SEHR zeitaufwendige Untersuchung der ArangoDB HTTP API vorausgesetzt

Ergebnisse

API

  • klar gegliedert in Namespaces
  • alle Methoden sind ähnlich aufgebaut
  • vielseitige Methoden
  • vollständig funktionales Arbeiten möglich
    • 
							(with-connection {:server-url "http://.."}
  ;; do something
  )
- gegliedert in wenige Namespaces, die deren Funktion klar anzeigen (weniger als bei ArangoDB) - z.B. mehrere Query Namespaces von ArangoDB zusammengefasst - Methoden sind gleich aufgebaut bzgl. Reihenfolge der Argumente; Inkonsistenzen der ArangoDB HTTP API wurden beseitigt
